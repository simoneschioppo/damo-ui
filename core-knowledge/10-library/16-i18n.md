# i18n (Internationalisation)

Status: documented · Last scan: 27c8471 · Sources:
`packages/ui/src/lib/i18n/{types.ts,provider.tsx,index.ts,dictionaries/{en,it,index}.ts,__tests__/provider.test.tsx}`.

## Summary

The library ships a tiny custom React-context i18n primitive so
consumers can render damo-ui components with locale-aware default
labels (loading text, dismiss buttons, calendar locale, pagination
chrome) **without** pulling in a Next- or framework-specific i18n
runtime. Default locale is **English**; Italian is the bundled
alternative.

The lib stays framework-agnostic on purpose: the apps/web docs site
adopts `next-intl` separately and bridges to the lib by mounting a
matching `<I18nProvider>` (see
[20-web-app/10-docs-site/README.md](../../20-web-app/10-docs-site/README.md)).

## Public API

| Export             | Kind                                       |
|--------------------|--------------------------------------------|
| `I18nProvider`     | React component (`'use client'`)           |
| `useI18n`          | React hook returning the active dictionary |
| `useLocale`        | React hook returning the active locale     |
| `dictionaries`     | `Record<Locale, Dictionary>` (en, it)      |
| `DEFAULT_LOCALE`   | `'en'`                                     |
| `Locale`           | `'en' \| 'it'`                             |
| `Dictionary`       | structural type — see below                |
| `I18nProviderProps`| `{ locale?, dictionary?, children }`       |

Re-exported from `packages/ui/src/index.ts`.

### `Dictionary` shape

```ts
interface Dictionary {
  spinner: { label: string }
  combobox: {
    placeholder: string
    searchPlaceholder: string
    emptyMessage: string
  }
  datePicker: {
    placeholder: string
    dateFnsLocale: Locale  // date-fns Locale object
  }
  pagination: {
    previous: string
    next: string
    page: string
    pageOf: (page: number, total: number) => string
  }
  banner: { dismissLabel: string }
  dialog: { closeLabel: string }
  drawer: { closeLabel: string }
  toast: { closeLabel: string }
}
```

Function entries (`pagination.pageOf`) are intentional — the v1
provider has no ICU runtime, so any interpolation is expressed as a
typed function.

## Internal architecture

### Provider

```tsx
const I18nContext = createContext<I18nContextValue | null>(null)

export function I18nProvider({ locale = 'en', dictionary, children }) {
  const value = useMemo(
    () => ({ locale, dict: dictionary ?? dictionaries[locale] }),
    [locale, dictionary],
  )
  return <I18nContext.Provider value={value}>{children}</I18nContext.Provider>
}

export function useI18n(): Dictionary {
  return useContext(I18nContext)?.dict ?? dictionaries.en
}

export function useLocale(): Locale {
  return useContext(I18nContext)?.locale ?? 'en'
}
```

### Fallback semantics — "no provider, no error"

`useI18n()` returns the **English** dictionary when no `<I18nProvider>`
is mounted. This matters because:

1. Library tests, Ladle stories, and external consumers without a
   provider must still render every component without throwing.
2. The lib commits to "EN-default for external consumers" — no
   surprise Italian leakage when a consumer drops `<Spinner />` into
   a fresh React tree.

The fallback is read-only: bare components render in EN even if some
ancestor has localized via plain `<html lang="...">`. To switch
locale, mount `<I18nProvider locale="it">` explicitly.

### Caller props always win

Every component reads its dictionary entry only when the caller did
not pass a value. Pattern (from `Spinner`):

```tsx
function Spinner({ label, ...rest }) {
  const i18n = useI18n()
  const resolvedLabel = label ?? i18n.spinner.label
  return <svg aria-label={resolvedLabel} {...rest} />
}
```

This is back-compat critical: any pre-i18n call site that passed an
explicit string (likely Italian, from the lib's pre-PR-69 defaults)
keeps rendering that exact string regardless of locale.

### `dictionary` prop — escape hatch

`<I18nProvider dictionary={...}>` lets consumers inject a custom
shape. v1 ships `en` and `it`; if a consumer needs `de`, `ja`, etc.
they can build a `Dictionary` and pass it directly:

```tsx
const customDe: Dictionary = { ...en, spinner: { label: 'Lädt…' } }
<I18nProvider locale="en" dictionary={customDe}>{children}</I18nProvider>
```

Note that `locale` and `dictionary` are independent — the locale
remains whatever was passed (or `'en'` default) while the dictionary
overrides what each component sees. This is good enough for v1; if
the lib later adds a `getMessage(key)` API the locale will become
the canonical store.

## Components consuming the dictionary

| Component       | Dictionary path                 | Caller prop |
|-----------------|---------------------------------|-------------|
| `Spinner`       | `spinner.label`                 | `label`     |
| `Combobox`      | `combobox.placeholder`          | `placeholder` |
| `Combobox`      | `combobox.searchPlaceholder`    | `searchPlaceholder` |
| `Combobox`      | `combobox.emptyMessage`         | `emptyMessage` |
| `DatePicker`    | `datePicker.placeholder`        | `placeholder` |
| `DatePicker`    | `datePicker.dateFnsLocale`      | (DayPicker `locale` prop) |
| `Pagination`    | `pagination.{previous,next,page,pageOf}` | `labels` (shallow-merged) |
| `Banner`        | `banner.dismissLabel`           | `dismissLabel` |
| `Dialog`        | `dialog.closeLabel`             | (`aria-label` on built-in close X — no override yet) |
| `Drawer`        | `drawer.closeLabel`             | (same as Dialog) |
| `Toast`         | `toast.closeLabel`              | (same as Dialog) |

The Dialog / Drawer / Toast close-button labels do not currently have
an explicit prop override; if you need a custom string, hide the
built-in close (`hideClose` on Dialog/Drawer) and render your own.

## Notes & gotchas

1. **No ICU runtime.** Function entries are the only interpolation
   pattern. Adding ICU later means promoting `Dictionary` from
   strings to message keys + a runtime; back-compat would require
   that the function shape keep working.
2. **`'use client'`.** Provider is a client component (`useContext`,
   `useMemo`). Library consumers in RSC can mount it inside their
   client tree.
3. **Date-fns locale is a runtime object**, not a string code. The
   dictionary holds the actual `Locale` import (`enUS` / `it` from
   `date-fns/locale`). Keeps DatePicker's import surface unchanged
   and avoids a string→object lookup in the component.
4. **Adding a new lib-shipped locale** (e.g. `de`):
   - Extend the `Locale` union in `types.ts`.
   - Add `dictionaries/de.ts`.
   - Update `dictionaries/index.ts`.
   - Smoke-test all 8 consuming components under the new provider.
5. **Adding a new dictionary key**:
   - Extend `Dictionary` in `types.ts`.
   - Update `en.ts` and `it.ts`.
   - Wire the consuming component to `useI18n().<path>` with a
     caller-prop fallback.
   - Add a smoke test under `components/<name>/<name>.i18n.test.tsx`
     covering EN / IT / no-provider / caller-prop-wins.

## How to consume (shadcn-style copy)

The lib's i18n module is a single small folder
(`lib/i18n/{types.ts,provider.tsx,dictionaries/}`). Three options:

1. **Lift wholesale** — copy `lib/i18n/` into your app and import
   `useI18n` from there. The simplest path; works as long as you
   also lift any component that reads from the dictionary.
2. **Stub** — replace `useI18n` with a constant returning your own
   dictionary. Useful when you've already lifted only one or two
   components and don't want a provider-context dependency.
3. **Adapt** — wrap your existing i18n stack (`next-intl`,
   `react-i18next`, etc.) so its hook returns a `Dictionary`-shaped
   object, and re-export it as `useI18n`. The consuming components
   never see the underlying runtime.

## Open questions

1. **No public `addLocale(code, dictionary)` helper.** v1 expects
   consumers to either ship in an in-tree dictionary import or pass
   `dictionary` per-provider-mount. A registry could simplify the
   "load `de.ts` once at boot" pattern.
2. **Dialog / Drawer / Toast close-button override missing.**
   `aria-label` resolves from the dict but there is no `closeLabel`
   prop on `DialogContent` / `DrawerContent` / `ToastClose`. Add for
   parity with `Banner.dismissLabel`.
3. **No "missing key" telemetry.** `useI18n` falls back silently to
   EN when no provider is mounted. There is no signal in dev when a
   consumer probably forgot to mount the provider. A `data-i18n=…`
   attribute or a one-time console.warn could catch this.
4. **`pagination.pageOf` cannot be overridden via the `labels`
   prop's `pageOf`** without re-implementing the entire function.
   That is OK today (back-compat for previous callers passing
   custom `pageOf`), but a future "labels accept partial templates"
   design could be cleaner.
